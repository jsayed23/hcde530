# Week 2 Reflection: Competency 2 (Code Literacy and Documentation)

## What I worked on
This week I practiced reading and writing beginner-friendly Python code that processes CSV data, then summarizes results in both terminal output and a simple web dashboard. I focused on understanding what each code block does and documenting logic in plain language.

## Key observations

### 1) I can follow data flow better when code is organized step-by-step
I noticed that my understanding improves when the script clearly separates:
- loading data,
- transforming data,
- calculating summary metrics,
- printing or displaying results.

When those stages are visible, I can explain the script more confidently and troubleshoot faster.

### 2) Documentation is most useful when it explains intent, not just syntax
I learned that comments are more helpful when they explain why a step matters (for example, why we truncate a preview or why we compute averages), not just what the line literally does.

This is important for my audience because many readers are not software engineers and need conceptual guidance.

### 3) Small formatting details improve readability a lot
Clear output formatting (aligned columns, short previews, labels) made it easier to scan results. I also saw how minor logic choices affect communication quality, such as only adding "..." when text is actually truncated.

## What this says about my code literacy
- I am getting more comfortable reading code.
- I can identify key logic decisions and explain their impact on output.
- I still need practice reducing repetitive comments.

## What this says about my documentation practice
- I should document the expected input format (CSV columns) and output behavior.
- I should prioritize "how to run this" instructions for mixed-skill classmates and instructors.
- I should keep one source of truth for project context and coding expectations.

## Next steps
1. Refactor comments to remove redundancy and keep only high-value explanations.
2. Add a short run guide to README with terminal and web preview steps.
3. Keep a consistent documentation pattern:
   - purpose,
   - inputs,
   - outputs,
   - key logic notes.

## Why this matters for UX practice
As a UX practitioner, code literacy helps me collaborate across design and engineering by making analysis workflows transparent, reproducible, and easier to critique. Strong documentation helps others trust and reuse the work.
